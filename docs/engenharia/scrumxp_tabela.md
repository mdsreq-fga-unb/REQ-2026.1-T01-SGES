# Engenharia de Requisitos e o ScrumXP

| Fases do Processo | Atividades Er | Prática | Técnica | Resultado Esperado |
| :---- | :---- | :---- | :---- | :---- |
| Planejamento da Release | Elicitação e Descoberta | Levantamento de Requisitos | Entrevistas (Vice-presidente/Diretores), Brainstorming e Análise de Domínio | Requisitos de alto nível identificados e compreensão dos gargalos da gestão manual. |
|  | Análise e Consenso | Priorização de Requisitos | Priorização MoSCoW e Análise de Custo/Benefício | Definição do MVP (Cadastro de beneficiários e Chamada digital) e viabilidade financeira. |
|  | Declaração | Registro dos Requisitos | Temas, Épicos e User Stories | Backlog inicial organizado em grandes blocos de impacto social. |
| Planejamento da Sprint | Elicitação e Descoberta | Refinamento de Requisitos | Entrevistas (Professores voluntários) e Análise Documental (Cadernos/Planilhas) | Refinamento dos detalhes operacionais e mapeamento de campos necessários para o digital. |
|  | Análise e Consenso | Análise de Dependências | Discussões em Equipe e Análise de Tarefas | Definição técnica da centralização dos dados e foco na usabilidade para voluntários. |
|  | Declaração | Definição de Critérios de Aceitação | Critérios de Aceitação Detalhados e Definition of Ready (DoR) | User Stories prontas para desenvolvimento com regras de negócio claras. |
|  | Organização e Atualização | Refinamento dos requisitos | Grooming do Backlog | Backlog refinado e alinhado às urgências atuais da instituição. |
| Execução da Sprint | Representação | Criação de Protótipos | Protótipos e Wireframes | Interfaces simples e intuitivas validadas visualmente antes da codificação. |
|  | Verificação e Validação | Validação de Requisitos | Checklist e Revisão de Critérios de Aceitação | Garantia de segurança, privacidade e cumprimento dos requisitos de dados sociais. |
|  | Organização e Atualização | Revisão do Backlog | Revisão do Backlog da Sprint (DEEP) | Ajustes rápidos no fluxo de trabalho perante complexidades imprevistas. |
| Revisão da Sprint | Verificação e Validação | Demonstração ao Cliente | Coleta de Feedback e Workshop de Requisitos | Demonstração de funcionalidades e validação da usabilidade real com os usuários. |
|  | Declaração | Atualização de User Stories | Incorporar Feedback e Negociação | User Stories ajustadas e priorizadas para facilitar a adoção do sistema. |
| Retrospectiva da Sprint | Análise e Organização | Revisão do Processo | Discussões em Grupo e Análise de Causas | Identificação de melhorias na comunicação interna e com a instituição. |
|  | Atualização do Processo | Ajustes no Workflow | Atualização do Workflow e Resolução de Conflitos | Processo refinado para lidar com a rotatividade de voluntários e desafios técnicos. |
| Planejamento da Próxima Release | Elicitação e Descoberta | Identificação de Novos Requisitos | Workshops e Análise de Domínio de Negócio | Identificação de novas necessidades, como relatórios para captação de recursos. |
|  | Análise e Consenso | Priorização Estratégica | Priorização MoSCoW e Mapeamento de Valor | Funcionalidades de alto impacto (ex: Dashboards) priorizadas estrategicamente. |
|  | Declaração | Definição de Épicos e Histórias | Criação de Épicos, User Stories e INVEST | Requisitos da nova etapa claramente definidos, valiosos e testáveis. |
|  | Organização e Atualização | Revisão do Backlog | Revisão do Backlog da Release (DEEP) | Backlog da release atualizado e base tecnológica preparada para as novas funções. |
